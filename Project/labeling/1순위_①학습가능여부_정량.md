# 1순위 ① — 입력 1280에서 YOLO26x-seg 학습 가능 여부 (정량)

> 확인 항목 ①을 숫자로 정리. 표기: **[검증]** 공식 문서 확인 / **[계산]** 검증값으로부터 단순 산술 / **[추정]** 경험·스케일링 기반(실측 필요). 작성 2026-06-04.

---

## 0. 결론부터 (숫자)

- **학습 가능**합니다. 1280은 무리한 크기가 아니라 **표준 고해상 tier**입니다.
- **메모리 추정**(YOLO26x-seg ≈ YOLO11x-seg 62M급 기준):
  - **batch 4 → 약 17~25 GB** → **A6000(48GB) 가능, node009(95GB) 여유**
  - **batch 8 → 약 33~45 GB** → A6000 빠듯, node009 가능
  - **batch 16 → 약 65 GB** → node009(95GB)만 가능
- **단서**: 우리 데이터는 한 장에 instance가 평균 **~200개**라, seg 마스크 손실 메모리가 더 붙어 **위 표보다 batch를 한두 단계 낮춰야** 할 수 있습니다.
- **실제 한계를 정하는 변수는 "1280이냐"가 아니라 → batch 크기 × instance 개수**입니다.

---

## 1. 입력 텐서 크기 [계산]

원본 3840×2160(16:9)을 `imgsz=1280`으로 넣으면, 긴 변을 1280으로 맞추고 stride(32) 배수로 패딩:

| 항목 | 값 |
|------|-----|
| 스케일 | 3840→1280 = **×1/3** (입력 1px = 원본 3px) |
| 입력 해상도(직사각, 기본) | **1280 × 736** (2160→720→패딩 736) |
| 입력 픽셀 수 | 942,080 px (640×640 대비 **2.30배**) |
| (참고) 정사각 옵션 | 1280×1280 = 640² 대비 **4.0배** |

### feature map / 마스크 grid (입력 1280×736 기준)
| 층 | stride | 크기 |
|----|--------|------|
| 마스크 prototype | /4 | **320 × 184** |
| 검출 P3 | /8 | 160 × 92 |
| 검출 P4 | /16 | 80 × 46 |
| 검출 P5 | /32 | 40 × 23 |
| loss 마스크 grid(mask_ratio=4) | /16급 | **80 × 46** |

→ 모델이 마스크를 학습/표현하는 "해상도"는 **prototype 320×184**, loss는 더 거친 **80×46**. (이게 1순위 Point 상한의 근거 — `1순위_Point수_검토.md` §3)

## 2. 모델 스펙 [검증, Ultralytics 공식]

**YOLO26x-seg (우리 타깃 모델, 640 기준):**

| 항목 | 값 |
|------|-----|
| 파라미터 | **62.8M** |
| FLOPs | **313.5B @640** |
| 기본 학습/벤치 입력 | **640** |
| mask mAP(50-95, e2e) | 47.0 |
| box mAP | 56.5 |
| 추론속도 | T4 TensorRT 16.4ms |

> ※ 62.8M·313.5B는 YOLO26의 **e2e(NMS-free) fused 모델 기준**(배포용). 학습 시 보조 헤드가 붙어 약간 더 큼(추정 영향 미미). YOLO26은 semantic seg loss + 멀티스케일 proto 모듈로 YOLO11 대비 **+3.7 mask AP**.

(참고) YOLO11-seg 라인업 — v1에서 쓴 m-seg가 anchor:

| 모델 | 파라미터(M) | FLOPs(B)@640 |
|------|-------------|--------------|
| yolo11m-seg | 22.4 | 113.2 |
| yolo11x-seg | 62.1 | 296.4 |

→ YOLO26x-seg(62.8M)는 YOLO11x-seg(62.1M)와 **사실상 동일 컴퓨팅 등급**.

## 3. 연산량(FLOPs) @1280 [계산]

FLOPs는 입력 픽셀 수에 비례 → 640 값 × 2.30(직사각) 또는 ×4.0(정사각):

| 모델 | @640 | **@1280 직사각(×2.3)** | @1280 정사각(×4) |
|------|------|------------------------|-------------------|
| yolo11m-seg | 113 | ~260 GFLOPs | ~453 |
| **YOLO26x-seg** | **313.5** | **~721 GFLOPs** | ~1,254 |

- 한 장 forward ~721 GFLOPs, 역전파 포함 ~2~3배 → **학습 1 step(1장)에 약 1.5~2조 FLOPs**.
- A6000(TF32 텐서코어 ~80 TFLOPS급)·RTX PRO 6000 모두 **충분히 처리 가능한 범위**(연산 자체는 병목 아님).

## 4. GPU 메모리 추정 [추정]

학습 메모리 = **고정비(파라미터+옵티마이저) + batch × 이미지당 활성값**

### 4-1. 고정비 [계산]
- YOLO26x-seg 62.8M × 16 byte(가중치+gradient+AdamW의 m,v) ≈ **~1.0 GB** (batch 무관)

### 4-2. 이미지당 활성값 [추정]
- @1280 직사각에서 forward+backward 합쳐 **장당 약 4~6 GB**로 추정(seg 헤드·~200 instance 포함).
- 근거 anchor: v1에서 **yolo11m-seg @1280 batch=4가 node009(95GB)에서 정상 학습**됨 → m급 장당 ~2GB대로 추정, x급은 그 2~3배.

### 4-3. batch별 메모리 추정표 (x-seg @1280 직사각)
| batch | 추정 메모리 | A6000(48GB) | node009(95GB) |
|:-----:|:-----------:|:-----------:|:-------------:|
| 1 | ~5~7 GB | ✅ | ✅ |
| 2 | ~9~13 GB | ✅ | ✅ |
| **4** | **~17~25 GB** | ✅ | ✅ (여유) |
| 8 | ~33~45 GB | ⚠️ 빠듯 | ✅ |
| 16 | ~65 GB | ❌ | ✅ |

> 모두 **추정치**입니다. 정확값은 `batch=-1`(Ultralytics AutoBatch)로 수 분 내 실측 가능 — 이게 가장 빠른 확정 방법입니다.

---

## 5. 우리 데이터 보정 — instance 수가 진짜 변수 [추정]

- COCO 평균 instance/이미지 ~7개인데 **우리는 ~200개**. seg는 instance마다 마스크 손실을 계산하므로 메모리가 추가됩니다.
- v1에서 `max_det=500`을 썼던 이유도 이것. → **위 표보다 batch를 한두 단계 낮추거나 max_det 제한**이 필요할 수 있음.
- **즉, "1280이라 안 된다"가 아니라, "1280은 되지만 batch·instance 수를 조절해야 한다"** 가 정확한 결론.

## 6. 한 줄 정리

| 질문 | 정량 답 |
|------|---------|
| 1280에서 YOLO26x-seg 학습 가능? | **가능** |
| 연산량 | 장당 ~721 GFLOPs(forward), GPU 처리에 병목 아님 |
| 메모리 (batch 4 기준) | **~17~25 GB 추정** → A6000(48GB)·node009(95GB) 모두 가능 |
| 진짜 한계 | **batch × instance(~200/장)** — 1280 여부가 아님 |
| 확정 방법 | `batch=-1` AutoBatch로 수 분 내 실측 |

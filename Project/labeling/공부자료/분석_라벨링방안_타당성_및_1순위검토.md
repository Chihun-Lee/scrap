## 1. 1순위 검토 — 입력 1280 기준 Polygon Point 수 (담당 ITV, 우리도 검토)

### 1.1 결정적 사실: YOLO-seg가 학습할 수 있는 마스크 디테일의 상한
**[F]** YOLO11/26-seg 마스크는 prototype(stride 4) 기반 → **imgsz 1280서 prototype grid 320×320**. loss는 `mask_ratio=4`로 한 번 더 다운샘플 → **약 80×80 grid에서 supervision**.
**[F]** 4K(3840폭)→1280 스케일 **s=1/3** → **입력 1px = 원본 3px**, **proto cell 1개(입력 4px) ≈ 원본 12px**.

**[I] 함의**: polygon vertex가 proto cell(원본 ~12px, 입력 ~4px)보다 촘촘하면 **학습 신호가 0** — 더 거친 loss grid(~80×80)에선 더더욱. 즉 **세밀한 vertex는 mask 품질을 못 올리고 저장·전처리 비용만 증가.**

### 1.2 방어 가능한 Point 수 기준 (학회 논문 근거)

| 기준 | 값 | 근거 |
|---|---|---|
| **per-instance hard ceiling** | **~128 vertex** | Deep Snake [F]: 128 saturate, **192는 오히려 성능 저하** |
| **전형적 필요 수** | **30–40** (대형), 소형은 **<10** | Polygon-RNN++/Curve-GCN 30–40 [F], PolarMask 36 [F] |
| **원칙적 per-instance 규칙** | **vertex ≤ ceil(둘레@1280 / 4)** | proto cell = 4 입력px [I] |
| **단순화 수단** | Douglas–Peucker `cv2.approxPolyDP`, **ε ≈ 3–4px@1280 ≈ 원본 ~10–12px** | RDP 표준 [F] + proto-cell [I] |

→ **per-image 총 point**: 별도 절대상한을 못 박기보다 **per-instance RDP(ε≈1 proto cell) + 128 cap**을 걸면 자연히 수렴. 거친 order 추정: ~200 instance × 평균 ~20–40 vertex ≈ **이미지당 4,000–8,000 point** 수준(현재 4K 세밀 라벨은 이보다 수배~수십배 많을 것 → 여기서 OOM/I-O 비용 발생).

### 1.3 ⚠️ 공학적 교정 — "Point 수"가 GPU OOM의 *주*동인이 아닐 수 있음
요청서는 "Point 수 많음 → GPU 메모리 과다"로 기술하나, YOLO-seg 학습 메모리의 실제 주동인은:
1. **이미지당 instance 수** (target mask 채널 수 · 매칭 비용) ← **가장 큼**
2. **마스크 grid 해상도** (∝ imgsz²)
3. **batch · 모델 크기 · imgsz**

**[I]** Polygon **vertex 수**는 주로 ① 라벨 파일 크기, ② dataloader CPU rasterization, ③ augmentation 변환 비용에 영향. GPU activation 메모리엔 instance 수·마스크 해상도가 더 지배적.
→ **결론**: Point 상한은 여전히 타당(품질 무손실 단순화 + 저장·전처리 절감 + 안전마진)하지만, **진짜 OOM 완화는 instance 수 폭증을 잡는 2·4순위(소형 컷오프)와 묶어야** 효과적. **1순위와 2·4순위는 독립이 아니라 연동.** 이 점을 ITV에 회신 시 짚는 게 좋음.

### 1.4 1순위 "확인 요청 사항" 직답
- **1280 YOLO26x-seg 학습 가능?** [I] instance 수·batch에 의존. point cap+소형 컷오프 병행 시 가능성 높음. 실측 필요.
- **OOM 안 나는 point 수?** [I] per-instance RDP+128 cap으로 bound. 절대 임계는 instance 수와 결합해 실측.
- **instance당 최대 point?** **128 hard, 실무 40–60 권장** [F/I].
- **이미지당 총 point?** RDP로 자연 수렴(≈수천 단위 목표) [I].
- **point 단순화별 mask 품질?** [I] ε ≤ 1 proto cell까지 IoU 손실 ~0, 초과 시 급락 예상 → ε–IoU 곡선 실험 권장.
- **point 단순화별 학습가능/GPU?** [I] vertex↓ → 라벨 I/O·전처리·augment 비용↓(주로), GPU activation 영향은 부차적.

## 2. 클래스 규칙 변경 타당성

### 2.1 ② 대표 클래스 병합 (예: 배관 vs 아연도금 강관 → 배관) — ✅✅ 강하게 타당
- **[F]** Northcutt 2021: 라벨 오류의 **지배적 원천 = fine-grained·혼동 쉬운 클래스 쌍**. 병합이 이 노이즈를 직접 제거.
- **[F]** EQL: 혼동 클래스 간 **cross-class 억제 gradient**가 tail을 붕괴시킴 → 병합이 학습측에서도 이득.
- **[F]** Cityscapes: 클래스 계층 + "확신 가능한 level에서 라벨"을 제도화한 선례.
- **ver1 연계**: `Unknown-Panel→structure steel` 식 외형 충돌 병합이 신호 모순을 일으켰음(VER1_SUMMARY). 형상 기반 대표 클래스는 **그 충돌을 줄이는 방향** — 단 "대표 클래스 정의표"가 외형 기준으로 일관돼야(아래 §6 권고).

### 2.2 ① Unknown 재정의 — ✅ 타당, 단 누락 케이스 존재
- 변경(식별되나 목록에 없는 경우만 Unknown)은 **LVIS unknown-set 의미론과 정합** [F].
- **⚠️ GAP**: 기존 방식의 (i) "특정 클래스로 보이나 **불확실**" case가 변경안에서 **명시적 행선지가 없음**. 문헌은 이를 명확히 구분:
  - *식별되나 목록 밖* → **Unknown(명명)** ← 우리 규칙 ✅
  - *불확실/결정불가* → **void/ignore 또는 difficult 플래그**(Cityscapes/VOC), 명명 클래스로 X
- → **권고**: "불확실" case 처리(대표 클래스 강제 추정? skip? ignore 플래그?)를 가이드라인에 별도 명시. 안 하면 annotator가 불확실 객체를 Unknown에 버려 노이즈 발생.

## 3. 운영 원칙 (additive-only: 기존 유지 + 누락 보완) — ✅✅ 타당
- **[F]** Yaman 2024: segmentation 노이즈 중 **"missing instance"가 가장 해로운 축** → 누락 객체 추가가 최우선이라는 판단 정당.
- **[F]** COCO staged(spot→segment): 찾기와 분할을 분리해 재작업 최소화 = additive 워크플로 정당화.
- **[F]** Decoupled classifier(Kang) 등 데이터 고정·재라벨 0 기법과 궁합 — "삭제·재작업 지양" 철학과 일치.
- **단서** [F]: 추가 라벨이라도 **경계 부주의는 비쌈**(Yaman) → 추가 객체도 마스크 경계 품질 기준 유지.

## 4. 2·4순위 — Instance 크기 컷오프 타당성 (우리 담당)

### 4.1 stride 수식 기반 권고 (3840×2160 원본, 입력 1280, s=1/3)

| 임계 | 원본 크기 | 의미 / 근거 |
|---|---|---|
| **Hard floor (학습 무의미)** | side **< ~12px** (~144px²) | < 1 mask cell(stride 4), 마스크 불가 [F 수식] |
| **Detection floor** | side **< ~24px** (~576px²) | < 1 detection cell(P3 stride 8) [F 수식] |
| **최소 학습 임계 (4순위)** | side ≈ **24–48px** | floor 위 + TinyPerson "<20px 저하" [F] |
| **"충분히 큼"(2순위 착수)** | side ≳ **~96px** (~9k px²) | 입력서 COCO-medium+ ; "소형 skip해도 무방"(Hosoya 2024) [F] |

- **ver1 정합** [F/I]: v1은 640서 8px floor=원본 48px → 절반 이상 소실. **1280으로만 가도 floor가 원본 24px로 절반 개선.**
- **⚠️ 구조적 대안** [F]: **SAHI tiling** 채택 시 s→~1 → 모든 floor가 ~3× 축소(hard ~4px, 착수 ~32px). ~200 dense instance엔 강력 후보 → 2순위 결론 전 tiling 여부 결정이 임계값을 좌우.
- **희귀 클래스** [F]: 전역 cutoff 대신 *낮은* 임계 + copy-paste oversample(Kisantal) — ver1 minority 붕괴 대응.

### 4.2 2순위(러프)↔4순위(정밀)는 같은 축
둘 다 "소형 instance 크기 컷오프"를 *먼저 거칠게(6/26) → 나중 정밀(9/4)* 잡는 것. 클래스별 차이(4순위)는 표의 단일 임계를 클래스별로 분화.

## 5. 3순위 — 입력 크기 × Point 수 조합 (우리 담당) — ✅ 타당 + 보완
- **[F]** 연산·메모리 ~제곱: 960 대비 **1280=1.78×, 1600=2.78×, 1920=4.0×**. mAP는 체감 감소(YOLOv4).
- **[F]** 1280은 표준 고해상 tier(YOLOv7 별도 모델). YOLO11 C2PSA가 소형·가림 일부 보완.
- **[I] 예상**: **1280–1600 band가 accuracy–latency knee** 근처. 1920은 ~4× 비용에 한계 이득 + batch 강제 축소.
- **⚠️ 필수 보완** [F]: (a) overall mAP뿐 아니라 **AP_small을 크기별로 따로 측정**(SNIP — scale를 network에 맞추는 게 raw 해상도보다 중요). (b) **~1280 SAHI tiling과 head-to-head** — 소형 recall 병목이면 tiling이 imgsz↑보다 우월할 수 있음.

## 6. 발견된 GAP & 보완 제안

1. **클래스 불균형 (최우선 GAP)** — 4개 연구항목에 없음. 1600:1(structure steel 129k vs plastic 79). **재라벨 없이** 가능한 처방:
   - loss: **Seesaw(instance-seg 전용)** 또는 Focal [F]
   - **decoupled classifier**(재라벨 0) [F]
   - **평가: AP_rare/common/freq 분리 보고**(LVIS) [F] — 안 하면 tail 성능이 안 보임
   → ITV에 "연구항목에 불균형 추가" 제안 권고.
2. **dense pile 처리** — 분해 불가 밀집 더미에 **COCO `iscrowd`/ignore 영역** 도입 [F]: exhaustive 개별 라벨 강제의 원칙적 대안, additive-only와 양립.
3. **가림/잘림 = modal 명시** [F]: 가시 영역만 + occlusion/truncation 플래그(VOC). amodal 금지(우리 목표는 가시 면적/카운트). → vendor 추측 차단.
4. **PQ 겹침 해소** [F]: panoptic은 마스크 겹침 금지 → instance→panoptic 변환(6,8번 스크립트)의 depth/confidence 순서가 PQ에 직접 영향. 변환 규칙도 가이드라인/실험에 포함.
5. **"대표 클래스 정의표"** — 외형(형상) 기준 결정 트리로 명문화(예: 단면 형상·길이비·표면). 경합 규칙이 annotator마다 다르면 노이즈 재발.

## 7. 다관점 종합 평가

| 관점 | 1순위 Point | 2·4 크기 | 3 해상도 | 클래스 규칙 | 운영 additive |
|---|---|---|---|---|---|
| **학습 신호** | proto-cell 이상 무의미 → cap 타당 | <12px 신호 0 | 1280서 소형 floor 절반↓ | 병합=신호 모순↓ | 누락 추가=신호 회복 |
| **메모리/비용** | I/O·전처리↓(GPU 부차적) | instance↓=메모리 주효과 | 제곱 비용, 1920 OOM 위험 | — | 재작업 비용↓ |
| **평가(PQ)** | 경계 단순화는 IoU>0.5 위협 안 함(ε≤cell) | 소형 FN↓ | AP_small 따로 봐야 | 클래스 균등 가중 유리 | FN↓→RQ↑ |
| **라벨 노이즈** | — | 모호 소형 노이즈↓ | — | 혼동쌍 제거(최대 노이즈원) | missing=최대 노이즈 제거 |
| **일정/실용** | RDP 자동 적용 간단 | 거친→정밀 단계화 부합 | knee 탐색 비용 큼 | 결정 트리 필요 | vendor 친화적 |

## 8. 우리(재료연구원) 담당 실험 권고 (2/3/4순위)

1. **2순위 (6/26, 러프 충분크기)**: ver1 데이터 클래스별 크기 분포(이미 `data/analysis/report.txt` 보유)에 §4.1 임계 적용 → "원본 side ≳ ~96px(또는 tiling 시 ~32px)" 1차 기준 + 적용 시 추가 라벨링 대상 instance 수 변화 산출.
2. **3순위 (8/4)**: {960,1280,1600,1920} × {point 단순화 수준} 격자로 Box/Mask mAP·**AP_small**·추론속도·GPU 메모리 측정 + **SAHI@1280 대조군**.
3. **4순위 (9/4)**: 2순위 임계를 클래스별로 분화 + 겹침/절단/구김/파손 객체 modal 유지 기준 확정 + 성능·효율 변화.
4. **(제안) 불균형**: Seesaw/Focal + stratified AP를 3·4순위 실험에 곁들여 보고.

> 모든 수치는 검증된 stride/연산 규칙 기반 **추론**이며, 프로젝트의 "러프 1차 기준 → 실험으로 단계 보완" 철학에 맞게 **우리 데이터 실측으로 확정**해야 함.

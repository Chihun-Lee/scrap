## 요약 (3줄)

1. **"1878픽셀 이미지"는 별도 생성 이미지가 아니라**, 6/23 공유(1024 기준) 실험에 쓰인 화물칸 크롭 이미지들의 **평균 긴 변(1,877.86px)** 으로, 리사이즈 임계값을 원본 픽셀로 환산할 때 쓴 상수입니다. 현행(1280) 기준에서는 사용하지 않습니다.
2. **현행 컷오프 기준은 이미 bbox가 아니라 폴리곤 면적 기준**입니다 — "8px@1280"은 √(폴리곤 면적)=8px, 즉 면적 64px²@1280(원본 576px², 이미지 면적의 0.007%)을 뜻합니다.
3. 요청하신 대로 **1280 입력 → stride 4 feature map(320×320)에서 GT 형상 유지 여부를 전수 시뮬레이션**한 결과: 완전 소실은 드물지만(√면적 4px@1280 이상 생존율 ~100%), **두께 원본 6px 미만 세장형은 GT가 절반 이상 조각나** 지적하신 "철근 소실" 현상을 정량 재현했습니다. 다만 재학습 검증(§5)에서 이 조각난 GT도 rebar 학습에 위치 신호로 기여함이 확인되어, **기본 컷 30px + 세장형 예외(길이 72px↑) 유지, 두께 6px 미만은 우선순위 최하위**를 최종안으로 제안합니다.

## 1. "1878 픽셀 이미지"의 정체 (확인요청 ①)

| 항목 | 내용 |
|---|---|
| 정체 | 별도 기준으로 생성한 이미지가 아님. **6/23 공유 실험(YOLO11, imgsz=1024)에 사용된 "Cargo Area 크롭 이미지"들의 평균 긴 변** |
| 생성 과정 | 원본 4K(3840×2160)에서 라벨 `73. Cargo Area` 폴리곤 영역만 크롭 → 크롭 결과는 이미지마다 크기가 다름 (Train 평균 긴 변 1,877.76px / Val 1,878.26px / 전체 1,877.86px) |
| 용도 | 1024 리사이즈 후 필터값의 원본 환산 상수. 예: 16px@1024 ÷ (1024/1877.86) ≈ 원본 29.34px |
| 현행 기준과의 관계 | **7/8 회신부터는 크롭 없이 4K 풀프레임 + 1280 입력 기준으로 전환** → 환산은 1280/3840 = 1/3 고정. 1878px는 더 이상 사용하지 않으며, 과거(1024) 실험과 새 기준을 잇는 환산에만 등장 |

즉 6/23 자료(1024, 크롭, 축소비 ≈0.545)와 7/8 자료(1280, 풀프레임, 축소비 0.333)는 **스케일 체계가 다르고**, 1878은 전자를 후자로 번역하기 위한 분모였습니다.

## 2. 컷오프 기준: bbox가 아니라 폴리곤 면적 (확인요청 ②)

- 7/8 공유한 분포·컷오프 수치의 크기 지표는 **√(Instance Polygon 면적) @1280** 입니다 (신발끈 공식으로 폴리곤 면적 계산 → 1280 스케일 환산 → 제곱근).
  - "8px 미만" = 면적 64px²@1280 = **원본 576px²** (이미지 전체 면적의 0.007%)
  - "10px 미만(제안)" = 면적 100px²@1280 = **원본 900px²** (0.011%)
- 오해의 소지가 있던 부분: 6/23 자료(1024 실험)의 필터는 bbox 짧은 변+면적 병용 기준이었고, "30×30px" 같은 표기가 변 길이처럼 읽힐 수 있었습니다. **현행 ver2 기준은 순수 면적 기준**이며, 정사각형 객체라면 한 변 ~30px에 해당한다는 직관적 표기였습니다.
- 지적하신 문제의식(긴 형상은 bbox로는 커 보여도 실제 픽셀이 적음)은 타당하며, 실제로 저희 분포 분석에서도 bbox 기준은 세장형이 많은 본 데이터에서 변별력이 없어 면적 기준을 채택했습니다.
- 참고로 COCO의 small/medium/large(32²/96²) 구분도 bbox가 아니라 **segmentation mask 픽셀 면적** 기준이며(cocoeval), 라벨링 하한의 해외 실무 사례로는 nuImages "높이 10px 미만 객체는 라벨링 금지", DOTA v1.0 "10px 미만 미라벨" 등이 있습니다.
- 단, 현행 기준에는 반대 방향의 **세장형 예외**(면적 미달이어도 bbox 긴 변 ≥ 원본 72px면 라벨 유지, 약 2.9만 개)가 있는데, §3 시뮬레이션 결과 이 예외를 **두께 조건과 결합**해야 함을 확인했습니다 → §4 개정안.

## 3. stride 4 feature map 기준 학습 가능성 — 전수 시뮬레이션 (확인요청 ③)

### 3.1 검토 기준의 정식화 (요청 기준 채택)

YOLO26x-seg의 마스크 경로를 코드 레벨로 확인했습니다 (ultralytics ≥8.4):

- 검출 헤드는 P3/P4/P5(stride 8/16/32). 마스크 prototype(`Proto26`)은 P4·P5를 P3 해상도로 융합 후 ×2 업샘플해 **stride 4** 해상도로 생성 → 1280 입력이면 **320×320×32** (풀프레임 4K는 letterbox로 1280×720 → 유효 proto 영역 320×180).
- **학습 seg loss도 같은 해상도에서 계산**: GT 폴리곤을 `fillPoly(입력해상도) → cv2.resize(×1/4)`로 다운샘플(`mask_ratio=4`, `polygon2mask`)한 마스크가 타깃. cv2.resize(INTER_LINEAR)의 uint8 반올림 특성상 **커버리지 ~50% 임계의 다운샘플**이라, 입력 기준 폭 4px(원본 12px) 미만 구조는 위상에 따라 GT에서 소실됩니다.
- 검출 측면에서는 YOLO26의 **STAL**(small-target-aware label assignment)이 입력 8px(원본 24px) 미만 박스도 후보 선택 시 16px로 가상 확대해 **positive 할당을 보장**합니다(공식 ablation: AP_small +0.6). 즉 YOLO26에서 소형 객체의 병목은 "검출 할당"이 아니라 **stride 4 마스크 표현**입니다.
- 추가 메커니즘: seg loss는 GT bbox 내부 픽셀 평균 BCE(crop_mask)라, 대각선 방향 세장형은 bbox 내 배경 픽셀이 압도해 얇은 전경의 학습 신호가 희석됩니다.
- 따라서 "해당 Instance가 학습 가능한가"의 1차 필요조건은 **stride 4 그리드에서 GT 형상이 유지되는가**이며, 이는 학습 없이 GT 생성 파이프라인 재현만으로 전수 측정이 가능합니다.

### 3.2 시뮬레이션 설계

- 전체 라벨 501,377 인스턴스(train 417,218 + val 84,159, Cargo 제외)에 대해 ultralytics GT 생성 파이프라인을 그대로 재현: 폴리곤 → 1280 letterbox 스케일 → fillPoly → ×1/4 다운샘플(INTER_LINEAR, stride-4 그리드 위상 정렬).
- 지표:
  - **생존율**: 320 그리드에서 픽셀 ≥1 남는 비율 (완전 소실 여부)
  - **복원 IoU**: 320 GT를 1280으로 되살렸을 때 원 마스크와의 IoU — stride 4에서 달성 가능한 마스크 품질의 이론적 상한
  - **조각남**: 320 GT가 2개 이상 연결성분으로 분리되는 비율 (긴 객체가 끊어지는 현상)

## 5. 학습 검증 결과 (2026-07-20 완료 — YOLO26x-seg @1280, 100ep, batch 4)

### 5.1 exp2 — 컷오프 스윕 (핵심: 공통 val 기준 판독)

| 학습 데이터 | 자기 val mask mAP50 | **공통 val(무필터) mask mAP50** | 무필터 대비 | 라벨 공수 |
|---|---:|---:|---:|---:|
| 무필터(base) | 0.4872 | 0.4872 | — | 100% |
| cut8 (원본 24px) | 0.6454 | **0.4857** | −0.15%p | 52% |
| **cut10 (원본 30px, 제안)** | 0.7043 | **0.4766** | −1.06%p | **46.5%** |
| cut12 (원본 36px) | 0.7259 | 0.4510 | −3.62%p | 42% |
| cut16 (원본 48px) | 0.7440 | 0.4301 | −5.71%p | 38% |

**판독**:
- ⚠️ **자기 val 착시의 실증**: 자기 val로는 컷을 높일수록 성능이 오르는 것처럼 보이지만(0.645→0.744, 과거 1024 스윕과 동일 패턴), 동일한 무필터 val로 재평가하면 **정반대**입니다. 지난 결과의 "필터 높일수록 개선"은 평가 라벨이 함께 걸러진 효과였습니다.
- **24~30px 구간이 최적**: 30px 컷은 무필터 학습 대비 −1.06%p에 라벨 공수 절반 이하, 24px 컷은 −0.15%p(사실상 동등)에 공수 52%. **36px 이상은 손실이 가속(−3.6%p~)되어 기각.**
- 무필터 학습(base)조차 0.4872로 cut8과 0.15%p 차이 — **30px 미만 소형 라벨의 실질 학습 기여는 미미**함을 재확인 (exp5 시뮬레이션과 일치).

### 5.2 exp6 — 세장형 예외 정책 (cut10 고정, 공통 val)

| 예외 정책 | mask mAP50 | rebar AP50 | small pipe AP50 | 라벨 수 |
|---|---:|---:|---:|---:|
| **현행: 길이≥72px 유지** | 0.4766 | **0.1055** | **0.0846** | 232,978 |
| 개정안: +두께≥6px 조건 | 0.4777 | 0.0815 | 0.0812 | 222,029 |
| 예외 없음 | 0.4801 | 0.0634 | 0.0623 | 204,135 |

**판독**:
- **세장형 예외는 필수**: 예외를 없애면 rebar AP가 40% 급락(0.106→0.063) — 전체 mAP +0.35%p로는 보상 안 됨.
- **§3.4 시뮬레이션 가설의 부분 수정**: 두께 6px 미만의 조각난 GT도 rebar 학습에 신호로 기여함이 확인됐습니다(현행 0.1055 > 두께조건 0.0815). GT 형상은 뭉개져도 "그 위치에 철근이 있다"는 위치 신호는 남는 것으로 해석됩니다.
- **권고**: 예외 규칙은 **현행(길이 원본 72px 이상 유지)을 기본**으로 하고, 두께 원본 6px 미만 세장형은 **우선순위 최하위(여력 시 라벨)**로 두는 것을 제안합니다. 단 세 정책의 전체 mAP 차이(0.35%p)와 rebar AP 차이는 단일 런 기준이라 반복 실험 분산과 겹칠 수 있음을 함께 말씀드립니다.

### 5.3 exp7 — 학습 설정 대안 (mask_ratio) 검증

| 설정 | mask mAP50(공통) | rebar AP50 | 비고 |
|---|---:|---:|---|
| mask_ratio=4 (기본, cut10) | **0.4766** | **0.1055** | GT를 320²로 다운샘플 |
| mask_ratio=2 (cut10) | 0.4644 | 0.0866 | GT 640², 두께 보존 하한 12→6px |
| mask_ratio=1 | **실행 불가** | — | 이미지당 [인스턴스 수×1280²] 마스크 텐서 → 인스턴스 수천 개 이미지에서 A100 80GB로도 OOM |

**판독**: GT 해상도를 높여도(mr2) 전체·세장형 모두 개선되지 않음 → **mask_ratio는 기본값(4) 유지**. "stride 4 GT 표현 한계"는 학습 설정 변경보다 §5.2의 라벨 정책과 (후속) 입력 크기 상향으로 다루는 것이 유효합니다.

rebar 계열은 §3.4상 1280의 구조적 한계가 있어, **입력 크기 상향 시 세장형 학습 가능 두께 하한이 어떻게 내려가는지**를 7/31 스크리닝에서 확인 예정. (exp2의 조건별 추론 속도는 7런 동시 실행 오염으로 이번 표에서 제외 — exp3에서 단독 측정.)
